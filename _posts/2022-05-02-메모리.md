# 메모리

메모리는 다른 장치와 용어상으로 혼동될수 있으나, 대체로 `주기억장치`를 말하며 특히 `RAM`을 말한다.

![mem_hierarchy](https://upload.wikimedia.org/wikipedia/commons/c/c6/%EB%A9%94%EB%AA%A8%EB%A6%AC%EA%B3%84%EC%B8%B5%EA%B5%AC%EC%A1%B0%EA%B7%B8%EB%A6%BC1.png)

다음과 같은 `memory hierarchy`에서 위로갈수록 빠르고 용량이 적고, 아래로 갈수록 용량이 크고 느리다. 우리가 프로그램을 실행할때 기록된 모든 내용이나 정보를 매번 디스크연산을 통해 가지고 오게 되면 매우 느리다. CPU가 조금더 빨리 데이터에 접근하기 위해 메모리를 사용한다.

## OS의 메모리 관리

고전적인 OS는 물리메모리 전체에 OS와 실행시킬 단 하나의 프로세스를 메모리에 올려서 프로세스를 실행했다.
 메모리의 크기도 크지 않았던 시절이지만 상상만 해도 엄청 느리고 비효율적이다.

![mem1](https://imgur.com/sXfzOSA.jpg)

 시간이 지난 후 여러 프로세스들을 물리메모리 상에 올려서 실행하는 방법이 나온다. 효율성이 올라갔지만 이때부터 `protection issue`가 생기기 시작했다. 
 현재 프로세스를 실행하다가 다른 프로세스의 메모리를 건드려 값을 수정하는등의 행위를 막아야 했다.
 ![mem2](https://imgur.com/IP6sbEU.jpg)

 이러한 이유로 OS는 `Address Space(주소 공간)`라는 물리메모리의 가상화를 만들었고, 이 주소 공간은 실행중인 프로세스의 모든 정보를 담고 있도록 했다.
 ![addr_space](https://imgur.com/0g9Es59.jpg)

 그리고 모든 프로세스는 `가상 메모리`를 사용하도록 하고, 실제 `물리 메모리`로의 translation은 `OS`가 담당한다. 하지만 소프트웨어 상에서 주소변환을 하면 너무 느리기 때문에, 하드웨어의 도움을 받아서 처리한다. (registers, TLB, page-table)

 하지만 실제로 주소공간은 가상적인 메모리이고, 물리메모리의 매핑되는 경우 가상메모리의 크기를 벗어나면 다른 프로세스 메모리에 접근할 수 있으므로 `Base and Bound Register`을 통해, 현재 접근하는 메모리의 주소를 계산해서 적법한 메모리 접근인지 판단한 후 실행한다.
 ![bbregister](https://imgur.com/4OWSQgO.jpg)

이러한 방법을 사용하면, `External fragmentation(외부 단편화)'`가 발생하게 되는데,

![ext](https://imgur.com/S7t1ZmY.jpg)

* 외부 단편화란
다음과 같이 프로세스의 크기가 제각각 다른데, 필요한 부분마다 할당되고 해제되다보니 마지막 부분의 남는 공간을 더하면 `Process 11`을 메모리공간에 할당할 수 있으나, 각각의 공간은 새로운 프로세스를 로드하기에 너무 공간이 작아, `Process 11`을 메모리에 올리지 못한다.

이에 대한 해결책으로 `Segmentation`을 사용하게 되는데, 프로세스의 메모리에는 `Code, Stack, Data Heap`이 있으므로 영역마다 나눠서 따로 메모리에 올리고, 각각의 `Base and Bound register`을 이용하여 위의 방법처럼 처리를 할 수있다. 하지만 여전히 외부단편화 문제가 발생하란, 이 영역을 압축시키는 `Compaction`은 비용이 너무 크다.(프로세스를 멈추고, data를 어딘가에 복사해두고 segment register값들을 새로운 주소로 바꾼다)

또한 Heap영역같이 동적으로 변하는 메모리에는 실제 사용하는 메모리보다 더 많은 메모리를 할당해야 하는데 이런 경우에
 `Internal framgentation(내부 단편화)`가 발생한다.

 * 내부 단편화란
 할당받은 공간에 비해 사용중인 공간의 크기가 작다면, 그 차이만큼은 노는공간(의미없는 공간)이 되고, 낭비하는 공간이 된다. 
 ex) 4096KB만큼의 주소공간을 할당받은 프로세스의 실제 사용하는 주소공간이 10KB일 경우 4086KB는 사용하지 않아 낭비된다.


 이러한 문제점들을 해결하기 위해 실행되는 프로세스 공간만 나누는게 아니라, 실제 물리메모리를 일정한 크기로 자르는 방법인 `Paging`이 제안된다.

프로세스를 일정한 크기인 `page`라는 고정된 크기로 자르고, 물리메모리도 동일한 `page`단위로 자른다.
이러면 두개의 메모리 크기가 동일하기 때문에 각 주소공간의 페이지번호와 물리메모리의 페이지번호를 `Page table`을 통해 매핑시켜주는 방법이 `Paging`이다. 이런 방법을 사용하면 쉽게 메모리 관리를 할 수 있고, 힙과 스택이 어디로 자라는지 등의 상세내용을 알 필요가 없다.
![addr_translate](https://imgur.com/ambhRkE.jpg)

이러한 매핑 작업을 하는 페이지 테이블도 물리메모리에 존재하게 되는데, 먼저 페이지테이블을 보고 실제 물리페이지의 주소를 알 수 있기때문에 느리다는 단점이 존재한다.
따라서 하드웨어 캐시를 이용하여, 페이지 주소변환을 빠르게 수행한다.
![tlb](https://imgur.com/iB52STm.jpg)
